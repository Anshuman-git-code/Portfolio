01 / 04 Production-Oriented Architecture

Production Serverless Event Ticketing Platform

Business problem — modern event platforms must support registration, ticket generation and email delivery while staying responsive under load. The objective was an event-driven serverless platform that handles registration synchronously and everything downstream of it — ticket generation, delivery — asynchronously, without making the attendee wait on either.

Engineering Decisions

Why EventBridge?

To decouple registration from ticket generation — the attendee gets a fast, guaranteed response, and ticket creation scales independently on its own schedule.

Why DynamoDB?

Access patterns were mapped before the schema — three tables with Global Secondary Indexes support every query the app needs without table scans.

Why Terraform?

Reusable, versioned Infrastructure as Code — the entire environment is reproducible from a single apply.

Why serverless?

Registration traffic is spiky around event announcements — Lambda gives automatic scaling with no idle operational cost.

API GatewayReceives and routes HTTP requests
LambdaRuns registration and ticketing logic
DynamoDBStores events, registrations, tickets
EventBridgeDecouples registration from ticketing
SESDelivers the generated ticket by email
S3Stores ticket files privately
CognitoJWT authentication & role access
TerraformProvisions the full infrastructure
Security

Authentication

Amazon Cognito issues JWT tokens; every API request is verified before it reaches business logic.

Authorization

Role-based access control distinguishes organizer and attendee capabilities inside the backend.

IAM

Each Lambda has its own role scoped to only the resources that function needs — least privilege by default.

Storage

Ticket files stay private in S3; access is granted only through short-lived pre-signed URLs, never public reads.

Production Considerations & Lessons

The hardest part wasn't wiring EventBridge — it was resisting the urge to make registration wait for ticket delivery. Once I accepted that registration and fulfillment are two different guarantees, the async boundary became obvious. The bigger lesson: identify the business capability the system revolves around before choosing any AWS service around it.

02 / 04 Production-Oriented SaaS

LinkForge Pro — URL Shortener & Analytics Platform

Business problem — organizations need a secure URL-shortening platform with branded short codes, click analytics, and administrative visibility — not just a redirect service. Link creation, redirection, and administration needed to scale and evolve independently of one another.

Engineering Decisions

Why separate capabilities?

Link creation, redirection and administration evolve at different rates — splitting them into independent Lambdas means each can change without risking the others.

Why prioritize redirect over analytics?

If a click-count update fails, the user is still redirected successfully — a broken analytics write should never become a broken link.

Why hash-key on short code?

Every redirect operation starts with the short code, so it made sense as the DynamoDB partition key for single-digit-millisecond lookups.

API GatewayExposes 3 independent REST APIs
LambdaOne scoped function per capability
DynamoDBHash-key lookup by short code
CloudFrontFast static frontend delivery
IAMLeast-privilege role per Lambda
CloudWatchLogs and operational metrics
Security

Validation

URL format, length limits, and custom short-code collisions are checked before any write.

IAM

Each of the three Lambdas holds a role scoped only to the DynamoDB actions it actually performs.

CORS

API Gateway CORS is origin-restricted rather than wildcarded.

Production Considerations & Lessons

The lesson from this project wasn't about implementing features — it was learning to identify independent business capabilities and give each one a single responsibility. That separation is what made the system easy to extend later with QR codes and CSV export without touching the redirect path.

03 / 04 AI + Serverless

Resume Parser & Job Matching Platform

Business problem — recruiters spend minutes manually reviewing every resume for skills and suitability, a process that doesn't scale with applicant volume. The goal was a pipeline where a recruiter uploads a PDF and the system extracts skills, matches candidates to jobs, and surfaces results automatically.

Engineering Decisions

Why an event-driven pipeline?

Four independent stages — upload, extract, parse, match — each with one responsibility, instead of one large service performing everything sequentially.

Why Textract over a custom parser?

Document intelligence at ~99% extraction accuracy is a solved problem — building that in-house wasn't where the engineering value was.

Why DynamoDB Streams?

Streams let each stage react to the previous one's write without polling, keeping the pipeline fully event-driven end to end.

TextractExtracts structured text from PDFs
LambdaRuns each pipeline stage
S3Stores raw & processed resumes
DynamoDBCandidate profiles & match scores
API GatewayRecruiter-facing REST APIs
TerraformRepeatable infrastructure
Security

Storage

Resumes are never public — access happens only through pre-signed S3 URLs.

IAM

Six Lambdas, six scoped roles — each touches only its own stage's resources.

Validation

Upload Handler validates file type and size before anything reaches the processing pipeline.

Production Considerations & Lessons

Large business workflows get dramatically easier once they're decomposed into independent, event-driven stages instead of one application trying to do everything. Each stage focuses on a single responsibility and hands off automatically — that's what let the pipeline process real candidates against real job postings without a single orchestration script.

04 / 04 Event Driven

Serverless Image Optimization Pipeline

Business problem — uploading and resizing high-resolution images synchronously slows down the upload request and wastes compute. The goal was letting users upload once while the system generates multiple optimized resolutions automatically, without Lambda ever sitting in the file-transfer path.

Engineering Decisions

Why keep Lambda out of the transfer path?

S3 handles object uploads far more efficiently than Lambda ever could — routing the file through compute would only add latency and cost.

Why two separate buckets?

Splitting original and processed images gives cleaner IAM boundaries, simpler lifecycle rules, and easier debugging than one shared bucket.

Why Pillow in a Docker-built layer?

Native image libraries need to match the Lambda execution environment — building via Docker guarantees Amazon Linux compatibility.

LambdaAuthorizes uploads, resizes images
S3Separate input & output buckets
API GatewayIssues pre-signed URL requests
CloudFrontDelivers processed images
IAMPer-Lambda least-privilege roles
Terraform30 resources, fully reproducible
Security

Storage

Both buckets stay private; every read and write happens through short-lived pre-signed URLs.

IAM

Each Lambda's role is scoped to exactly the bucket and actions its stage needs — nothing shared.

Production Considerations & Lessons

Good cloud architecture isn't about moving data through more services — it's about moving data as little as possible while moving permissions intelligently. Letting storage handle storage and compute handle processing made the system more scalable and far easier to reason about.

Engineering Philosophy

How I think as a Cloud Engineer

I believe cloud engineering begins with understanding the business problem, not selecting AWS services. Every project in this portfolio was designed by deriving architecture from business requirements, identifying capabilities and ownership boundaries, evaluating trade-offs, and only then choosing the technology that satisfies them.

In practice, that means working through actors, business capabilities, and access patterns before a single AWS resource gets provisioned. A DynamoDB table gets its keys from the queries the application actually needs to run — not the other way around.

I focus on understanding why a system should be designed a certain way before deciding how to implement it. That order — not the reverse — is what makes systems easier to maintain, scale, and reason about six months later, and it's the order every project on this page follows.

  1. 01

    Business Requirement

    What actual problem is this solving, and for whom?

  2. 02

    Architecture Design

    Actors, business capabilities, ownership boundaries and access patterns — mapped before any service is chosen.

  3. 03

    AWS Services

    Services are selected to satisfy the architecture — never the other way around.

  4. 04

    Infrastructure as Code

    Every resource defined in Terraform or CloudFormation, versioned and reproducible.

  5. 05

    Security

    Least-privilege IAM, private storage, and authentication designed in, not bolted on.

  6. 06

    Verification

    Does the system actually behave the way the architecture intended?

  7. 07

    Deployment

    Shipped through a repeatable pipeline, not a manual console click-through.
